
## 0. メタ

* 文書種別: 改修要件定義(EARS 記法)
* 対象: Mermaid 画像変換 API(`/render`)
* 親文書: `開発.要件定義_Mermaid画像変換API要件定義書初稿MVP改訂版.md`
* 関連: `API仕様_Mermaid画像変換API.md`
* 起票背景: `craft/themes/nsc-internal/progress-reports/2026-05-10-01/` で本 API を利用して配布用 HTML を生成中、`flowchart` の日本語ラベルが `foreignObject` 内で見切れる事象が発生し、本 API 側でユーザー側コードに手を入れずに解消する手段が無いことが判明したため。
* 作成日: 2026-05-10

---

## 1. 目的

* 利用側が Mermaid コードを編集せずに、**ノード内テキストの見切れを抑止**できるようにする。
* リクエストごとに**ノード余白(nodePadding)**および**ダイアグラム余白(diagramPadding)**を制御できるようにする。
* レスポンス SVG に対して、**インライン `max-width` の上書き**を任意で行えるようにする(配布用 HTML への埋込時、レスポンシブ崩れを回避するため)。
* 既存の `MERMAID_PADDING` 環境変数(SVG ルートの外周パディング)との後方互換を維持する。

---

## 2. 背景・問題定義

### 2.1 現象

* `flowchart` で日本語ラベル(CJK)+複数行+ノード幅制約の組合せにおいて、ノード内 `foreignObject` の内側で**テキスト下端 / 右端が見切れる**。
* 例: 「集める ✓<br>(PrimeDrive 自動)」を `flowchart LR` のノードに入れると、ラベルの一部が枠線にめり込む。

### 2.2 直接原因(調査結果)

* Mermaid の flowchart は各ノードを `<foreignObject>` でレンダリングし、内部の `<div>` の `width` を**ラベル測定値ベースで決定**する。CJK の `word-break` 計算と Mermaid の幅推定にズレが出ると、`foreignObject` の高さ・幅がテキストの実描画サイズより小さく確定し、はみ出した部分が clip される。
* 既存の `MERMAID_PADDING`(`config.ts:24`)は `svg { padding: ${padding}px; }` を `themeCSS` に注入するのみで、**SVG の外周にしか効かない**。ノード単位の余白(`flowchart.nodePadding` / `flowchart.diagramPadding`)には介入していない。

### 2.3 ユーザー影響

* 配布用 HTML(進捗報告 / 提案書 / Note 記事) における視認性が低下。
* 利用者は本来不要な「Mermaid コード側の試行錯誤」(改行位置調整 / `<br>` 除去 / テキスト短縮)を強いられる。
* API 側に手を入れない場合、利用者ごとに同じワークアラウンドが繰り返される。

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| `foreignObject` | SVG 内に HTML を埋め込むための要素。Mermaid は flowchart のラベル描画に使用する |
| `nodePadding` | Mermaid `flowchartConfig` のキー。ノード内側のテキスト余白(px) |
| `diagramPadding` | Mermaid `flowchartConfig` のキー。ダイアグラム全体の外周余白(px) |
| `themeCSS` | Mermaid 設定。レンダリング時に SVG にインジェクトされる CSS 文字列 |
| ルート余白 | 既存 `MERMAID_PADDING` が制御する `<svg>` 自体の `padding`(外周空白) |
| ノード余白 | `flowchartConfig.nodePadding` が制御する各ノード内側の空白 |

## 4. ユーザーストーリー

### US-01 (配布資料作成者)

> **As a** 進捗報告 / 提案書を Mermaid + 単一 HTML で配布する作成者
> **I want** 日本語複数行ラベルがノード内で見切れずに描画される
> **So that** 後加工(SVG 編集 / 文言短縮 / レイアウト試行)なしに、最終的な見栄えのまま配布物に埋め込める。

* 受け入れ条件:
  - `flowchart LR` で `<br>` 改行を含む CJK ラベル(例: 「集める ✓<br>(PrimeDrive 自動)」)を送信したとき、ノード内に**全文字が完全に収まる** SVG が返る。
  - 利用側はリクエストパラメータ 1 つで挙動を切り替えられる。

### US-02 (同上)

> **As a** 進捗報告作成者
> **I want** リクエストごとにノード余白の強さ(コンパクト / 標準 / ゆったり)を変えたい
> **So that** 同一プロジェクト内で「俯瞰図はゆったり、詳細図はコンパクト」のように使い分けられる。

* 受け入れ条件:
  - リクエストボディに `flowchart_config.node_padding` 等の数値を指定したとき、その値が反映された SVG が返る。

### US-03 (運用 / 既存利用者)

> **As an** 既存 API 利用者
> **I want** 本改修の前後で、**何もリクエストを変えない場合の出力が壊れない**
> **So that** 既存の自動化スクリプトがそのまま動き続ける。

* 受け入れ条件:
  - 既存の最小リクエスト(`code` + `format` のみ)に対するレスポンスが、本改修前と**バイト互換**で出力される(=後方互換)。
  - 既存環境変数 `MERMAID_PADDING` の挙動は不変。

### US-04 (HTML 埋込配布)

> **As a** SVG を単一 HTML に埋め込んで配布する作成者
> **I want** レスポンス SVG に付与される `style="max-width: …px"` を、リクエスト時に剥がせる
> **So that** HTML 側の `max-width: 100%; height: auto;` でレスポンシブ表示できる(現状はクライアント側で正規表現置換している)。

* 受け入れ条件:
  - リクエストで `svg_post_process.strip_inline_max_width: true` を指定したとき、レスポンス SVG ルート要素から `max-width` インラインスタイルが除去されている。

## 5. スコープ

### 5.1 In Scope

* `flowchart` 系図種における `nodePadding` / `diagramPadding` のリクエスト時上書き
* レスポンス SVG に対する任意の後処理(MVP は `strip_inline_max_width` のみ)
* リクエスト互換性の維持(新パラメータは全て optional)
* 既存ドキュメント(`API仕様_Mermaid画像変換API.md`)への追記

### 5.2 Out of Scope

* `flowchart` 以外の図種(sequence / state / er 等)固有の余白パラメータ対応(必要になった時点で別票)
* Mermaid `themeCSS` の任意文字列注入(セキュリティ評価が別途必要)
* SVG 後処理での「テキスト幅自動再計算」(過剰実装。利用者は `nodePadding` で調整可能)
* キャッシュ / 認証 / 課金(MVP 親文書の Out of Scope 維持)

## 6. 機能要件(EARS 記法)

### 6.1 ユビキタス要件(常に)

* **REQ-U-10**: システムは リクエストボディの `flowchart_config.node_padding` を**受け付け、Mermaid `flowchartConfig.nodePadding` として適用しなければならない(SHALL)**。
* **REQ-U-11**: システムは リクエストボディの `flowchart_config.diagram_padding` を**受け付け、Mermaid `flowchartConfig.diagramPadding` として適用しなければならない(SHALL)**。
* **REQ-U-12**: システムは リクエストボディの `svg_post_process.strip_inline_max_width` を**受け付け、SVG ルート要素のインライン `max-width` を除去しなければならない(SHALL)**。
* **REQ-U-13**: システムは 既存環境変数 `MERMAID_PADDING` の挙動を**変更してはならない(SHALL NOT)**。

### 6.2 イベント駆動要件(〜したとき)

* **REQ-E-10**: クライアントが `flowchart_config.node_padding` に**正の整数**を指定したとき、システムは Mermaid 設定 `flowchart.nodePadding` をその値で**上書きしなければならない(SHALL)**。
* **REQ-E-11**: クライアントが `flowchart_config.node_padding` に**0 以下の値 / 非整数 / 文字列**を指定したとき、システムは HTTP 400 を返却し、`error_type=invalid_request` としなければならない(SHALL)。
* **REQ-E-12**: クライアントが `flowchart_config` 全体を**指定しなかった**とき、システムは Mermaid のデフォルト挙動を維持しなければならない(SHALL)。
* **REQ-E-13**: クライアントが `svg_post_process.strip_inline_max_width: true` を `format=png` と組み合わせて指定したとき、システムは**当該パラメータを無視**し、ログに `ignored_post_process` を出力しなければならない(SHALL)。
* **REQ-E-14**: クライアントが `flowchart_config` を `flowchart` 以外の図種(例: `sequenceDiagram`)と組み合わせて指定したとき、システムは**エラー化せず、当該パラメータを無視**しなければならない(SHALL)。

### 6.3 状態駆動要件(〜の間)

* **REQ-S-10**: 同一リクエスト内で `flowchart_config.node_padding` と環境変数 `MERMAID_PADDING` の両方が有効な間、システムは**両者を独立に適用**しなければならない(SHALL)。`MERMAID_PADDING` は SVG 外周、`node_padding` はノード内余白に作用する。
* **REQ-S-11**: SVG 後処理を実行している間、システムは元 SVG のバイト列を**ログに出力してはならない(SHALL NOT)**(機微情報混入防止)。

### 6.4 望ましくない挙動

* **REQ-UN-10**: システムは Mermaid `themeCSS` への**任意文字列注入**を行ってはならない(SHALL NOT)。
* **REQ-UN-11**: システムは リクエストパラメータの**未知のキー**を黙って受け入れてはならない(SHALL NOT)。未知のキーは検証時に警告ログに記録する。

## 7. API 変更案

### 7.1 リクエスト追加フィールド(全て optional)

```json
{
  "code": "...",
  "format": "svg",
  "timeout_ms": 8000,
  "flowchart_config": {
    "node_padding": 20,
    "diagram_padding": 16
  },
  "svg_post_process": {
    "strip_inline_max_width": true
  }
}
```

| フィールド | 型 | デフォルト | 範囲 | 説明 |
|---|---|---|---|---|
| `flowchart_config.node_padding` | integer | (Mermaid デフォルト=8) | 1〜200 | ノード内側の余白(px) |
| `flowchart_config.diagram_padding` | integer | (Mermaid デフォルト=8) | 0〜200 | ダイアグラム外周余白(px) |
| `svg_post_process.strip_inline_max_width` | boolean | `false` | — | SVG ルートのインライン `max-width` を除去 |

### 7.2 内部実装方針(参考)

* `config.ts` の `generateMermaidConfig(padding)` を `generateMermaidConfig({ rootPadding, nodePadding, diagramPadding })` に拡張(後方互換: 引数 1 個の旧呼び出しは廃止せず維持)。
* リクエストごとに Mermaid CLI に渡す設定 JSON を一時ファイル化(既存パターンの延長)。
* SVG 後処理は `renderer/mermaidRenderer.ts` のレスポンス組立直前で `format=svg` のときだけ適用。

### 7.3 エラーレスポンス例

```json
{
  "request_id": "01J…",
  "error_type": "invalid_request",
  "status_code": 400,
  "stderr": "flowchart_config.node_padding must be a positive integer between 1 and 200",
  "exit_code": null,
  "format": "svg"
}
```

## 8. 非機能要件

* **NFR-10**: システムは 新パラメータの**バリデーション失敗を mmdc 起動前に検出**しなければならない(SHALL)。理由: mmdc 起動コストを浪費しないため。
* **NFR-11**: システムは 新パラメータが指定されたリクエストについて、**処理時間ログに `extra_config=true` フラグを記録**しなければならない(SHALL)。理由: 改修後のパフォーマンス影響を観測可能にするため。
* **NFR-12**: SVG 後処理(`strip_inline_max_width`)の実行コストは**1 リクエストあたり 5ms 以内**でなければならない(SHALL)。

## 9. 受け入れ基準(AC)

* **AC-10**: `flowchart_config.node_padding=20` を指定して「集める ✓<br>(PrimeDrive 自動)」を含むノードを送ると、ノード内に**全文字が見切れずに**描画された SVG が返る。
* **AC-11**: `flowchart_config` を指定しないリクエストに対するレスポンスが、本改修**前と同一バイト列**で返る(後方互換)。
* **AC-12**: `flowchart_config.node_padding=0` を送ると HTTP 400 + `error_type=invalid_request` が返る。
* **AC-13**: `svg_post_process.strip_inline_max_width=true` を `format=svg` で送ると、レスポンス SVG ルートに `max-width` インラインスタイルが**含まれない**。
* **AC-14**: `svg_post_process.strip_inline_max_width=true` を `format=png` で送ると、PNG が**通常通り返る**(無視されたことが warning ログに記録される)。
* **AC-15**: `flowchart_config` を `sequenceDiagram` のコードと組み合わせて送っても、**エラーにならず**通常の sequence SVG が返る(無視ログ記録)。
* **AC-16**: 環境変数 `MERMAID_PADDING=40` の状態で `flowchart_config.node_padding=20` を送ると、SVG 外周は 40px パディング、ノード内は 20px パディングで**両方が独立適用**される。

## 11. 検討除外事項(意思決定の記録)

* **任意 `themeCSS` 注入**: セキュリティ評価コストが本改修の主訴(ノード見切れ)解消に対して過大なため除外。将来別票で扱う。
* **SVG レイアウト自動再計算**: 過剰実装。`node_padding` を増やすことで実用上同等の効果が得られる。
* **`useMaxWidth` 制御**: HTML 埋込時に SVG ルートの `max-width` を扱うのは `svg_post_process` 側で対処する方が**配布物側の表現力に対して直交**で素直なため、Mermaid 側 `useMaxWidth` の制御は採用しない。
